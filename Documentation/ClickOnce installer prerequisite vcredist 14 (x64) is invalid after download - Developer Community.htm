      ClickOnce installer prerequisite vcredist 14 (x64) is invalid after download

      Closed - Fixed visual studio 2019 version 16.1windows 10.0Fixed In: Visual Studio 2019 version 16.3.9

      Mikkel Hesselager Blann reported Sep 17, 2019 at 11:11 AM

      What steps will reproduce the problem?

      1. Create a new application (ie. WPF)
      2. Setup ClickOnce installer in project properties (publish)
      3. Add prerequisite to "Visual C++ 14 Runtime Libraries (x64)
      4. Select Download prerequisites from the component vendors web site
      5. Build and deploy project from Azure Devops, using hosted Azure Pipelines build agent
      6. Download the resulting setup.exe and run it on a computer without that vcredist

      The installer prompts to install the missing vcredist, but fails when the downloaded file is different from what was expected. The relevant part of the log file is pasted below.

      I tried using both VS2017 in vs2017-win2016 and VS2019 in windows-2019 hosted build agents, same problem currently.

      Every once in a while, we experience this problem again, without any real solution.

      Are the files at the versioned url being updated, so the package hash no longer matches the one from the build server? It seems impossible to get this to work reliably on a hosted build agent, where we cant manipulate the package files in the C:\Program Files (x86)\Microsoft SDKs\ClickOnce Bootstrapper folder.

      We might host the redist files ourselves, and specify a specific download location, but then well never know when the manifest actually gets changed on the build server, and we will need to update the files on our mirror.

      How are we supposed to work with MS prerequisites on hosted Azure Pipelines build agents?

      Curtis Smith (WINC) [MSFT] Oct 14, 2019 at 07:06 PM

      Thank you for reporting this issue we are working to get this fixed in upcoming releases.

      To workaround this issue:

      • Download the VC Redist packages you wish to use using the link in C:\Program Files (x86)\Microsoft SDKs\ClickOnce Bootstrapper\Packages\vcredist_[x86|x64]\en\package.xml
      • Right click the package and choose Properties
      • Select the Digital Signatures tab
      • Select one of the certificates listed
      • Select Details
      • Select View Certificate
      • Select the Details tab
      • Scroll down and select Public key
      • In the lower control select all of the data (starting with 30 82 01...) and copy
      • Paste the data into a text editor and remove all spaces
      • Open an Explorer window to C:\Program Files (x86)\Microsoft SDKs\ClickOnce Bootstrapper\Packages\vcredist_[x86|x64]
      • Right click product.xml and choose Properties
      • Select the Security tab
      • Select Edit
      • Select your user name and set to Modify
      • Select Ok
      • Select Ok
      • Open product.xml in a text editor
      • Copy the data and paste into product.xml replacing all data at PublicKey="308201..."
      • Save file

      Publish the project and test.

      croyston@sanjole.com Jan 10 at 08:52 PM

      As of Visual Studio 2017 v 15.9.18 and Visual Studio 2019 v 16.4.2 (released December 17, 2019) the packages dstributed in ClickOnce BootStrapper\Packages for both vcredist_x64 and vcredist_x86 still contained the wrong public key. I still had to update the public key manually to match Microsoft's current binary signing key.

      I've also noticed that different revisions of the package definitions in product.xml have used two different methods for detecting the presence of the run-time, which I had to choose between. Initially the packages used a registry check of HKLM\SOFTWARE\Microsoft\VisualStudio\14.0\VC\Runtimes\x64 (which is actually in the registry under WOW6432Node, to further confuse things):

      <RegistryCheck Property="VersionFullCheck" Key="HKLM\SOFTWARE\Microsoft\VisualStudio\14.0\VC\Runtimes\x86" Value="Version" />

      then at one point (I believe with VS 2017 15.9.15 and later) switched to:

      <MsiProductCheck Property="VCRedistInstalled" Product="{65835E57-3712-4382-990A-8D39008A8E0B}"/>

      and then switched back to using the registry check for the version. I don't know what problem there may have been with the MSI product ID check, but the problem with the registry check is that the redistributable package will not remove that key if uninstalled, so all this really checks is whether the given version was ever installed, not whether it's currently installed. This is probably a rare case but it came up when I was testing to validate my fix for the packages.

      I attach an example for x64 of the revised product.xml product-x64.txt and package.xml package-x64.txt which I settled on yesterday, which I expect to be using at my employer until this gets officially fixed. (While I was at it I also corrected the product description to Microsoft's canonical name for the runtime, rather than using "14" which seems to be purely an internal numbering scheme.)

      package-x64.txt (1.0 kB)

      croyston@sanjole.com No, I tried everything Jan 10 at 01:21 AM

      I have tried this fix after updating to Visual Studio 2017 15.9.18, and for good measure updating both to Visual Studio 2017 15.9.18 and to Visual Studio 2019 16.4.3 on a second machine.

      This update to the Click Once Bootstrapper package definitions does not solve the issue in the slightest and is virtually guaranteed to fail. While the package.xml file in the redist_x64 (and redist_x86) package definitions has been updated to the URL for the current revision of the runtime, which is good, and is now using a potentially stable (versioned) URL which is a very important improvement, it appears to me that the public key value in product.xml which the package expects to be used for code signing was never updated to match the public signing key Microsoft is currently using for binaries, dated May 5 2019-May 5 2020.

      As a result, the downloaded Visual C installer file will always fail validation; I don't see any way that an installer built with these options could ever succeed in installing the Visual C redistributable runtime.

      I find it very hard to believe that Microsoft tested this problem before reporting it fixed.
